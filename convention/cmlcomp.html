1. Introduction

The CMLComp convention is used to specify computational chemistry documents. It is intented to capture the typical underlying processes of quantum calculations and their relationships in a well defined implicit semantic structure using Chemical Markup Language (CML).

This document describes the concepts which are introduced in CMLComp, explains how to compose a CMLComp document and illustrates it with examples. The core concepts that make up CMLComp are:

  • Job list
  • Job
  • Initialisation
  • Iteration or calculation
  • Finalisation

The CMLComp convention also relies on the molecular convention which defines a set of constraints for storing to representations of molecules in CML. CMLComp requires that any molecules e.g. in the input molecular geometry and the optimised geometry result conform to the molecular convention. The molecular convention is not explained here and the reader should consult the molecular convention specification for more information.

1.1 Notational Conventions

The keywords "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 [ IETF RFC 2119 ].

The terms "element", "attribute", "child" and "parent" in this document are to be interpreted as described in the W3C Recommendation for Extensible Markup Language (XML) [ W3C XML ].

The use of fonts is as follows:

  • Schema terms, including elements and attributes, are written in this font.

2. Applying the CMLComp convention on CML element

The CMLComp document MUST contain a root cml element and the CMLComp convention, i.e. convention:cmlcomp, MUST be specified using the convention attribute on the cml element. 

                <?xml version="1.0" encoding="UTF-8" ?>
                <cml xmlns="http://www.xml-cml.org/schema"
                xmlns:convention="http://www.xml-cml.org/convention/"
                convention="convention:cmlcomp">

                <!-- document body is omitted. -->
                
                </cml>

If the convention is specified on a cml element then that element MUST contain at least one module child elements (with a role='joblist' attribute), and MUST NOT contain any other child elements in any namespace, see Example 1.

Example 1: Minimal requirements for CML element in CMLComp convention 

                <?xml version="1.0" encoding="UTF-8" ?>
                <cml xmlns="http://www.xml-cml.org/schema"
                xmlns:convention="http://www.xml-cml.org/convention/"
                convention="convention:cmlcomp">

                <module role="joblist">
                <!-- 1st job list body is omitted. -->
                </module>

                <module role="joblist">
                <!-- 2nd job list body is omitted. -->
                </module>

                </cml>

Example 2: CMLComp document MUST contain at least one module[@role="joblist"] 

                <?xml version="1.0" encoding="UTF-8" ?>
                <cml xmlns="http://www.xml-cml.org/schema"
                xmlns:convention="http://www.xml-cml.org/convention/"
                convention="convention:cmlcomp">

                <!-- Empty body is invalid CMLComp document. -->
                </cml>

3. CMLComp Concepts and Elements

3.1 Joblist

A quantum chemistry calculation is often comprised of a series of subtasks, e.g. coarse optimisation -> fine optimisation -> NMR Spectrum Analysis; this is because most quantum chemistry software packages are designed to be modularised and only to perform a single task at a time. The joblist concept is introduced to capture these series of successive subtasks and links the information from one subtask to the next subtask.

The job list can be defined by a cml:module where the role attribute MUST be set to joblist, i.e. cml:module[@role="joblist"]. This is shown in Example 1.

The job list module is a REQUIRED element and there MUST be at least one module element with a role attribute which MUST have the value "joblist". (See Example 3.)

Example 3: Minimal requirements for CMLComp Joblist 

        <module role="joblist">
        <module role="job" title="geometry optimisation">
        <!-- job body is omitted. -->
        </module>
        </module>

3.2 Job

The job concept (<module role='job'>) represents a computational job performed by quantum chemistry software, e.g. geometry optimisation job, frequency analysis job. The job concept is the smallest unit which can fully describe a general picture of computational modelling.

A job consists of model parameters, model optimisations or calculations and model results. These three components are fundamental to modelling in every field. However, a job in CMLComp does not require all three components. Only model parameters are mandatory.

A job is be defined by a cml:module where the role attribute MUST be set to "job" and it MUST conform with the following rules:

3.2.1 Job Title

The job module MUST have a title attribute

  • The CMLComp Job is defined by a cml:module which value of role attribute MUST be set to job.
  • The CMLComp Job MUST be titled using title attribute with unique string identifier within its job list module, i.e. cml:module[@role="joblist" and @title="unique ID"]. The job title only serves as human readable text and machine distinguishable identifier.
  • There MUST be at least one CMLComp Job in any CMLComp Joblist.
  • There MUST contains exactly one module of initialisation.
  • The CMLComp Job MAY contains at least one of calculation or optimsation module after the initialisation module stated in previous bullet point.
  • The CMLComp Job MAY contains exactly one finalisation module after the calculation or optimsation modules (if present) stated in previous bullet point or after the initialisation module (if calculation or optimsation modules are not present).
  • If the calculation or optimsation modules is present, CMLComp Job MUST contain a finalisation module.
  • The finalisation module MUST be the last element in CMLComp Job.
  • TODO: other elements allowed are identifier.

A CMLComp Job which contains only the initialisation module can be used for storing program or model inputs. It can be considered as an XML version of program input file.

Example 4: Minimal requirements for CMLComp Job module 

        <module role="job" title="coarse geometry optimisation">
        <module role="init">
        <!-- requried init body is omitted. -->
        </module>
        </module>

Example 5: CMLComp Job module with multiple calculation steps 

        <module role="job" title="coarse geometry optimisation">
        <module role="init">
        <!-- requried init body is omitted. -->
        </module>
        <module role="calculation">
        <!-- requried calculation body is omitted. -->
        </module>
        <module role="calculation">
        <!-- requried calculation body is omitted. -->
        </module>
        <module role="calculation">
        <!-- requried calculation body is omitted. -->
        </module>
        <module role="final">
        <!-- requried final body is omitted. -->
        </module>
        </module>

Example 6: CMLComp Job module without calculation steps 

        <module role="job" title="coarse geometry optimisation">
        <module role="init">
        <!-- requried init body is omitted. -->
        </module>
        <module role="final">
        <!-- requried final body is omitted. -->
        </module>
        </module>

3.3 Init

An initialisation module represents the concept of the model parameters and inputs for computational job. The initialisation module is mandatory in CMLComp. Here is a list of requirements:

  • The CMLComp Init is defined by a cml:module which value of role attribute MUST be set to init.
  • The CMLComp Init MAY contains exactly one cml:molecule element in molecular convention, i.e. molecular convention MUST be specified in molecule element.
  • The CMLComp Init MAY contains exactly one cml:parameterList element.
  • The CMLComp Init MUST contains at least one of cml:molecule, cml:parameterList or user defined module element.
  • There MUST be exactly one of CMLComp Init in a given CMLComp Job.
  • There MUST NOT be any other elements than given above in CMLComp Init except for one or more of user defined module.

Example 7: CMLComp Init module 

        <module role="init">
        <molecule convention="convention:molecular">
        <!-- consult molecular convention and molecule section below. -->
        </molecule>
        <parameterList>
        <!-- parameterList body is omitted. -->
        </parameterList>
        </module>

3.4 Calculation

A calculation module represents the concept of the model calculation or optimisation or iteration processes for computational job. The calculation module is optional in CMLComp. Here is a list of requirements:

  • The CMLComp Calculation is defined by a cml:module which value of role attribute MUST be set to calculation.
  • The CMLComp Calculation MAY contains exactly one cml:molecule element in molecular convention, i.e. molecular convention MUST be specified in molecule element.
  • The CMLComp Calculation MAY contains exactly one cml:parameterList element.
  • The CMLComp Calculation MAY contains exactly one cml:propertyList element.
  • The CMLComp Calculation MUST contains at least one of cml:molecule, cml:parameterList, cml:propertyList or user defined module element.
  • There MAY be zero or more than one of CMLComp Calculation in a given CMLComp Job.
  • There MUST NOT be any other elements than given above in CMLComp Calculation.

Example 8: CMLComp Calculation module 

        <module role="calculation">
        <molecule convention="convention:molecular">
        <!-- consult molecular convention and molecule section below. -->
        </molecule>
        <parameterList>
        <!-- parameterList body is omitted. -->
        </parameterList>
        <propertyList>
        <!-- propertyList body is omitted. -->
        </propertyList>
        </module>

3.5 Final

A finalisation module represents the concept of the model results for computational job. The finalisation module is optional in CMLComp. Here is a list of requirements:

  • The CMLComp Final is defined by a cml:module which value of role attribute MUST be set to final.
  • The CMLComp Final MAY contains exactly one cml:molecule element in molecular convention, i.e. molecular convention MUST be specified in molecule element.
  • The CMLComp Final MAY contains exactly one cml:propertyList element.
  • The CMLComp Final MUST contains at least one of cml:molecule, cml:propertyList or user defined module element.
  • There MUST be exactly one of CMLComp Final in a given CMLComp Job.
  • There MUST NOT be any other elements than given above in CMLComp Final.

Example 8: CMLComp Final module 

        <module role="final">
        <molecule convention="convention:molecular">
        <!-- consult molecular convention and molecule section below. -->
        </molecule>
        <propertyList>
        <!-- propertyList body is omitted. -->
        </propertyList>
        </module>

4. Data Structure with CML Elements

CMLComp has adopted a set of elements from CML to store the contents of the computational data. This section describes how CMLComp constructs the data structure from these CML elements set.

4.1 Molecule

CMLComp uses CML Molecule to store molecular geometry and coordinates. CML Molecule specifications are defined in molecular convention so the convention of the molecule element MUST be specified using the convention attribute with the value of convention:molecular. Readers SHOULD consult the specifications of molecular convention for more details.

4.2 z-Matrix

In many quantum chemistry calculations, the coordinates of the atoms are represented using z-Matrix coordinate system. CMLComp adopted z-Matrix from CML schema. (I have never used z-Matrix so I have very little experience. However, this is an important element and must be part of the CMLComp)

4.3 ParameterList

CMLComp uses CML ParameterList to group the CML Parameter(s) together. CML ParameterList MUST have only the CML Parameter(s) as child(s).

4.4 PropertyList

CMLComp uses CML PropertyList to group the CML Property(s) together. CML PropertyList MUST have only the CML Property(s) as child(s).

4.5 Parameter

CMLComp uses CML Parameter to represent the concept of the inputs of the quantum chemistry calculations.

A CML Parameter is defined by cml:parameter element and MUST have a referrence to an entry in a dictionary which specifies the meaning of the parameter.

A parameter MUST contain only one of the value containers given in section 4.8.

4.6 Property

CMLComp uses CML Property to represent the concept of the outputs of the quantum chemistry calculations.

A CML Property is defined by cml:property element and MUST have a referrence to an entry in a dictionary which specifies the meaning of the property.

A property MUST contain only one of the value containers given in section 4.8.

4.7 Units

Units are one of the most important information in scientific data. All value containers, see section 4.8, MUST have units associated with them except that they are string or text values.

  • The unit on a value container MUST be specified using unit attribute and the value MUST be QName of an entry in a unit dictionary. The unit dictionary MUST be provided along with the CMLComp document if the unit entry used in the CMLComp document does not exist on xml-cml.org.
  • TODO: Define units for dimensionles quantity.

4.8 Value container

CML provides elements to wrap many differrent types of mathematical, scientifical and computational values and variables, e.g. scalar, vector, matrix, array, etc., which will be addressed as value container in the current context. In CMLComp, we current only allow the use of scalar, array and matrix as value containers. This should be sufficiently enough for many computational outputs. Other type of CML containers will be added later.

4.8.1 Scalar

A CML Scalar element in CMLComp is used to hold a scalar data, which can be a single value of integer or real. The CML Scalar is also extended to include string or text value.

In CMLComp, the CML Scalar MUST conform with the following rules :

  • CML Scalar is defined by a cml:scalar element and MUST conform with CML schema version 3.
  • The data type of scalar is REQUIRED and MUST be specified using a dataType attribute. The value of dataType attribute MUST be one of xsd:string, xsd:integer or xsd:double. TODO: Decide whether to include other types, e.g. xsd:date, xsd:real, xsd:float.
  • scalar MUST have unit if the dataType is either xsd:integer or xsd:double even if they are dimensionless.
  • scalar MUST NOT have unit and unit type if the dataType is xsd:string.

4.8.2 Array

A CML Array element in CMLComp is used to hold a one dimentioanl array data structure of only either integer or real values, i.e. a collection of integer of real values where each element is identified by index. Thus, the data type of the CML Array MUST only be xsd:integer or xsd:double. It MUST also conform with the following rules :

  • CML Array is defined by a cml:array element and MUST conform with CML schema version 3.
  • The size of array is requried and MUST be specified using size attribute with the minimum value of 1.
  • TODO: delimiter, not sure what should be put here, the schema does not allow white space for dilimeter but cmlxom does.
  • The data type of array is REQUIRED and MUST be specified using dataType attribute. The value of dataType attribute MUST be one of xsd:integer or xsd:double. TODO: Decide whether to include other types, e.g. xsd:real, xsd:float.
  • array MUST have unit even if they are dimensionless.

4.8.3 Matrix

A CML Matrix element in CMLComp is used to hold a two-dimensional rectangular matrix data structure of only either integer or real values. Thus, the data type of this value container MUST only be xsd:integer or xsd:double. It MUST also conform with the following rules :

  • CML Matrix is defined by a cml:matrix element and MUST conform with CML schema version 3.
  • The dimension of matrix is requried and MUST be specified using rows and columns attributes with the minimum values of 1. (In CML Schema, this value is xsd:nonNegativeInteger which includes zero. Maybe this should be changed to xsd:positiveInteger)
  • TODO: delimiter, not sure what should be put here, the schema does not allow white space for dilimeter but cmlxom does.
  • The data type of matrix is REQUIRED and MUST be specified using dataType attribute. The value of dataType attribute MUST be one of xsd:integer or xsd:double. TODO: Decide whether to include other types, e.g. xsd:real, xsd:float.
  • matrix MUST have unit even if they are dimensionless.
